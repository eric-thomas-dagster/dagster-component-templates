# Supabase Vector Search

Run a [pgvector](https://github.com/pgvector/pgvector) similarity search against a Supabase table with a `vector` column. Reads a DataFrame of query embeddings from an upstream asset (one search per row) and returns matched rows with rank + query context.

## Installation

```bash
pip install "supabase>=2.0.0" "pandas>=1.5.0" "numpy>=1.20.0"
```

Requires a `SupabaseResource` (see `resources/supabase_resource`).

```yaml
type: dagster_component_templates.SupabaseVectorSearchAssetComponent
attributes:
  asset_name: doc_matches
  resource_name: supabase_resource
  table_name: documents
  embedding_column: embedding
  upstream_asset_key: query_embeddings
  query_embedding_column: embedding
  top_k: 10
  metric: cosine
  rpc_name: match_documents
```

## Canonical `match_documents` RPC

supabase-py's REST-based `.table()` client can't call pgvector's `<=>` operator directly. The recommended path is a Postgres function:

```sql
create or replace function match_documents (
  query_embedding vector(1536),
  match_count int,
  match_threshold float default 0
)
returns table (id bigint, content text, similarity float)
language sql stable
as $$
  select
    d.id,
    d.content,
    1 - (d.embedding <=> query_embedding) as similarity
  from documents d
  where 1 - (d.embedding <=> query_embedding) > match_threshold
  order by d.embedding <=> query_embedding
  limit match_count;
$$;
```

Then in YAML: `rpc_name: match_documents`.

For **L2** distance use `<->`; for **inner-product** use `<#>` (negate for a similarity score). Match the `metric` field to the operator in your RPC.

## pgvector literal format

Query vectors (from your upstream DataFrame or `query_vector`) can be either `list[float]` or `numpy.ndarray`. They are serialized to pgvector's text form `[0.1,0.2,0.3,...]` before being sent as the `query_embedding` RPC argument â€” pgvector parses this string form on the server side, so no client-side vector type is needed.

## REST fallback

If you leave `rpc_name` unset, the component uses `client.table(...).select(...).order(embedding, ...).limit(top_k)`. This does **not** perform a similarity search â€” PostgREST can't inject pgvector operators. The fallback simply orders rows by the raw `embedding_column` values and emits a warning. Create the RPC above for real similarity search.

## Output

Returns a DataFrame with one row per (query, match). Columns:
- `_query_row_index`: index of the query row in the upstream DataFrame
- `_rank`: 0-indexed rank of this match for its query
- All RPC-returned columns (e.g. `id`, `content`, `similarity`)
- `query_<col>` for every non-embedding column on the upstream row, for join-back
