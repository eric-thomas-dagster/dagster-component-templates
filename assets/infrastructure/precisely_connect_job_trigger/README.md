# precisely_connect_job_trigger

## Purpose

Materializable Dagster asset that triggers a Precisely Connect ETL
job-run via the `/runtask` endpoint. Use this when **Dagster owns the
schedule** — it fires Precisely, captures the returned `jobRunId` as
asset metadata, and (optionally) pairs with `precisely_job_sensor` in
observation mode to record terminal status on the same asset.

## Two ways to model a Precisely job in Dagster — pick ONE

| Case | Who owns the schedule | Components on this `asset_key` | Materialization timeline |
|---|---|---|---|
| **A. Observe-only** | Precisely | `external_precisely_job` (declare-only) + `precisely_job_sensor` (`asset_event_type=materialization`) | Sensor emits AssetMaterialization on terminal SUCCESS |
| **B. Dagster-triggered** | Dagster | **`precisely_connect_job_trigger`** (this) + optional `precisely_job_sensor` (`asset_event_type=observation`) | Trigger emits materialization at `/runtask` submit; sensor emits AssetObservation when Precisely confirms terminal SUCCESS |

**Do NOT mix `external_precisely_job` and `precisely_connect_job_trigger` on the same `asset_key`** — they're alternatives. Same `asset_key` value across the components in your chosen Case is the glue.

## Example YAML

```yaml
type: dagster_community_components.PreciselyConnectJobTriggerComponent
attributes:
  asset_key: precisely/etl/load_customers
  job_id: "abc-123-xyz"
  host_env_var: PRECISELY_HOST
  api_token_env_var: PRECISELY_API_TOKEN
  parameters:
    INPUT_FILE: /data/customers.csv
    BATCH_DATE: "2026-05-27"
```

Pair with a sensor in observation mode if you want the terminal status
to land on the same asset:

```yaml
# defs/precisely_job_sensor/defs.yaml
type: dagster_community_components.PreciselyJobSensorComponent
attributes:
  sensor_name: precisely_load_customers_done
  # Customers can set job_run_id from the latest materialization metadata
  # of the trigger asset — see Notes below for the dynamic-run-id pattern.
  job_run_id: "abc-123-def-456"
  asset_key: precisely/etl/load_customers     # SAME key as the trigger
  asset_event_type: observation               # NOT materialization
  host_env_var: PRECISELY_HOST
  api_token_env_var: PRECISELY_API_TOKEN
  job_name: precisely_downstream_job
```

## /runtask response shape

Connect ETL 5.x returns JSON `{"jobRunId": "..."}`. 4.x returns plain
text. This component handles both — `precisely_job_run_id` lands in
asset metadata either way (when parseable).

## Validation level
