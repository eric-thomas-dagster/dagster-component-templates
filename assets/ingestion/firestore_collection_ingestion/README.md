# Firestore Collection Ingestion

Read a Firestore collection (or subcollection) into a pandas DataFrame as a materializable Dagster asset.

## Installation

`pip install firebase-admin pandas`

Pair with a `firebase_resource` component in the same Dagster project.

## Configuration

```yaml
type: dagster_component_templates.FirestoreCollectionIngestionComponent
attributes:
  asset_name: active_users
  resource_name: firebase_resource
  collection_path: users
  where_clauses:
    - field: status
      op: "=="
      value: active
  order_by:
    field: created_at
    direction: desc
  limit: 1000
```

## Where Clauses

Each entry is a `{field, op, value}` dict. Supported ops mirror Firestore's
`FieldFilter`:

`==`, `!=`, `<`, `<=`, `>`, `>=`, `in`, `not-in`, `array-contains`, `array-contains-any`

```yaml
where_clauses:
  - field: tier
    op: in
    value: [gold, platinum]
  - field: signup_date
    op: ">="
    value: "2024-01-01"
```

## Subcollections

Firestore treats a path with an odd number of segments as a collection. Use a
literal id in the parent doc segment(s):

```yaml
collection_path: orders/ORD-12345/items
```

For querying items across *all* parents, use Firestore's collection group
queries (not supported directly by this component — pair with a wrapper).

## Output

The materialized DataFrame contains one row per document. The document id is
included in the `_id` column. Missing fields across documents become `NaN`
per pandas' default behavior for `DataFrame(list_of_dicts)`.

## Authentication

Configured on the paired `firebase_resource` component — this asset reads
`context.resources.<resource_name>` at runtime and never touches credentials
directly.

## Required IAM Roles

Grant the service account one of:

- `roles/datastore.viewer` — read-only access
- `roles/datastore.user` — read + write (only needed if other components in
  the same project write back to Firestore)
