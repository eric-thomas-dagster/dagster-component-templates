# Inline DataFrame

Materialize a literal pandas DataFrame straight from `defs.yaml` ‚Äî no CSV, no external file.

**Why this exists:** For small reference tables (region codes, currency conversions, fixture data, Alteryx **Text Input** ports), bundling a CSV alongside the project is heavier than the data itself. Drop the rows inline and you get an asset that's diff-reviewable in git.

Pairs with the `alteryx-to-dagster` migration tool ‚Äî Alteryx **Text Input** tools auto-route to this component instead of being emitted as a custom `@dg.asset` `.py` file.

---

## Install

```bash
dagster-component add inline_dataframe --auto-install
```

(only `pandas` ‚Äî no other deps.)

---

## Example

```yaml
type: dagster_component_templates.InlineDataframeComponent
attributes:
  asset_name: region_codes
  columns: [region_code, region_name]
  rows:
    - [N, North]
    - [S, South]
    - [E, East]
    - [W, West]
  group_name: reference
```

### With explicit dtypes

Useful when YAML values come in as strings (e.g. from an Alteryx Text Input XML port) but downstream filters need numeric comparisons:

```yaml
type: dagster_component_templates.InlineDataframeComponent
attributes:
  asset_name: raw_sales
  columns: [region, product, quantity, unit_price]
  dtypes: {quantity: Int64, unit_price: float64}
  rows:
    - [North, Widget, "10", "5.50"]
    - [North, Gadget, "3",  "12.00"]
    - [South, Widget, "7",  "5.50"]
```

Numeric dtypes (`Int64`, `Int32`, `float64`, etc.) route through `pd.to_numeric(errors="coerce")` so empty strings and `None` coerce cleanly. Other dtypes go through `.astype()`.

## When NOT to use this

| Use case | Right component |
|---|---|
| Reference table > ~100 rows | `dataframe_from_csv` (keep the CSV in the repo) |
| Data lives in a binary `.yxdb` | `dataframe_from_yxdb` |
| Data fetched at runtime from an API | `rest_api_to_dataframe` |
| Anything that needs to change without a code review | a managed data source ‚Äî Snowflake / Postgres / S3 |
