# GeoBufferComponent

Expand each geometry by `distance` units, producing a polygon buffer around
each input feature. Drop-in for Alteryx's **Buffer** and **Trade Area** tools.

For meaningful distances in real-world units (meters / miles), the component
projects from a geographic CRS (`EPSG:4326`, lat/lon in degrees) to a
projected CRS (default: `EPSG:3857` Web Mercator) before buffering, then
projects back. Web Mercator is accurate enough for most analytics; for
survey-grade accuracy, set `projected_crs` to the appropriate UTM zone or
state plane for your region.

## Use Cases

- Trade-area analysis ("which customers live within 5 miles of this store?")
- Site selection ("how much land is within 1 km of a planned facility?")
- Spatial filtering — buffer points of interest, then `geo_overlay` against
  another layer
- Coverage gap detection — buffer existing facilities and find the
  uncovered area via `geo_overlay` (difference)

## Example

  asset_name: store_trade_areas
  upstream_asset_key: store_locations_geo

## Notes

- `units` accepts `meters`, `km`, `miles`, `feet`, `yards`. The value is
  converted to meters before applying via the projected CRS.
- `resolution=16` (default) means 16 segments per quarter-circle. Decrease
  for faster math + smaller geometries; increase for smoother curves.
- Inputs without a CRS are assumed `EPSG:4326`.
- Mixed-CRS inputs are reprojected to `projected_crs` before buffering, so
  outputs are always consistent.
