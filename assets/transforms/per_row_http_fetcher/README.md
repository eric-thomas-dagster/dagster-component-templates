# PerRowHttpFetcherComponent

For each row in an input DataFrame, fire an HTTP request whose URL comes
from `url_column`, and append the response to that row. Drop-in for
Alteryx's **Download** tool (which also does per-row HTTP).

Adds three output columns (prefix configurable via `output_prefix`):

- `<prefix>_body` (`str`) — full response text
- `<prefix>_status` (`int`) — HTTP status code (`-1` on connection error / timeout)
- `<prefix>_json` (`dict | None`) — parsed JSON when `parse_json: true` AND
  the response content-type looks like JSON; `None` otherwise

Concurrent fetches via `concurrent.futures.ThreadPoolExecutor`
(`max_workers`), so a few hundred URLs are usually fast.

## Use Cases

- Enrich an address / domain list with public API data (e.g. WHOIS,
  weather, IP geolocation)
- Crawl product pages from a list of URLs and persist the raw HTML
  alongside metadata
- Hit a per-row webhook for downstream processing
- Resolve permalinks / shortened URLs in bulk
- Smoke-test a list of endpoints (status-code monitoring)

## Example

```yaml
type: dagster_community_components.PerRowHttpFetcherComponent
attributes:
  asset_name: enriched_product_pages
  upstream_asset_key: product_urls
  url_column: url
  method: GET
  headers:
    User-Agent: dagster-data-pipeline/1.0
    Accept: application/json
  timeout_seconds: 30
  max_workers: 20
  retries: 2
  output_prefix: response
  parse_json: true
  group_name: web_enrichment
```

## Notes

- For POST / PUT, request bodies aren't yet per-row — open an issue if you
  need this. v1 supports method + static headers per-call only.
- `retries` uses exponential backoff (1s, 2s, 4s, ...) and only retries on
  5xx + connection errors.
- `max_workers` is the size of the ThreadPoolExecutor — tune for the
  target service's rate limits. Default `10` is safe for most public APIs.
- Bodies are kept in memory; for very large responses use a smaller
  `max_workers` or chunk the input upstream via `sample`.
- Set `parse_json: false` if you're fetching binary content (the body
  column is still str — encoding is best-effort).
