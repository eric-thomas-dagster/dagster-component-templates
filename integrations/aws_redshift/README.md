# AWS Redshift

Imports AWS Redshift stored procedures and materialized views as Dagster assets using the Redshift Data API. Materialization executes the stored procedure (`CALL`) or refreshes the materialized view (`REFRESH MATERIALIZED VIEW`).

## Required packages

```
boto3>=1.26.0
botocore>=1.29.0
```

## Asset Dependencies & Lineage

Because this component enumerates many assets from one config, dependencies are declared per-asset via `asset_overrides` (keyed by the emitted asset's name). Matches the pattern used by the official [`DatabricksWorkspaceComponent`](https://docs.dagster.io/integrations/libraries/databricks/databricks-workspace-component#managing-dependencies).

```yaml
attributes:
  cluster_identifier: my-cluster
  database: analytics
  asset_overrides:
    procedure_daily_rollup:
      depends_on:
        - raw_orders              # simple asset key
        - raw/schema/orders       # slash-delimited ‚Üí hierarchical AssetKey
    matview_customer_summary:
      depends_on:
        - feature_store/customer_features
```

The declared deps draw lineage edges in the Dagster asset graph ‚Äî no data is loaded at runtime. Use them to express that a specific imported Redshift stored procedure or materialized view depends on upstream tables or assets produced elsewhere in your project. Assets without a matching entry in `asset_overrides` are emitted with no upstream deps (the default).

Dependencies can also be wired externally via `map_resolved_asset_specs()` in `definitions.py` ‚Äî the same approach used by [Dagster Designer](https://github.com/eric-thomas-dagster/dagster_designer).
