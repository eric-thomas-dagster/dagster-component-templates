# victoriametrics_snapshot_job

Create a [VictoriaMetrics](https://victoriametrics.com/) snapshot (point-in-time, hardlink-cheap) via the server's `/snapshot/create` endpoint, optionally with retention-based cleanup of older snapshots. The component produces a Dagster job ‚Äî optionally on a cron schedule ‚Äî that customers can wire into their backup flow.

VM snapshots are the canonical way to back up vmsingle / vmstorage before a destructive op (retention change, downgrade, schema migration) or as a scheduled offsite copy. The endpoint takes O(seconds) regardless of database size ‚Äî it's just a hardlink farm ‚Äî so a daily snapshot job adds zero ongoing cost.

## Example

```yaml
type: dagster_component_templates.VictoriaMetricsSnapshotJobComponent
attributes:
  job_name: vm_daily_snapshot
  schedule: "0 3 * * *"
  base_url_env_var: VM_BASE_URL
  retention_hours: 168    # keep 7 days
```

## Output metadata

Each run emits these op-output keys:

- `vm/snapshot_id` ‚Äî the new snapshot's ID (string)
- `vm/snapshots_deleted` ‚Äî count of retention-deleted snapshots
- `vm/snapshots_kept` ‚Äî count of snapshots kept (incl. the new one)
- `vm/deleted_ids` ‚Äî list of deleted snapshot IDs

## How retention works

When `retention_hours` is set, the op:

1. Creates a new snapshot.
2. Lists all existing snapshots via `/snapshot/list`.
3. Parses each ID's embedded timestamp (`YYYYMMDD[T]HHMMSS[Z]-<hash>`).
4. Deletes any older than now ‚àí `retention_hours`, leaving the new one + everything younger than the cutoff.

Unparseable IDs are kept (conservative). The new snapshot is always kept regardless of retention.

## Auth modes

- **No auth** ‚Äî fine for local Docker / private subnet.
- **`-snapshotAuthKey`** ‚Äî set `snapshot_auth_key_env_var: VM_SNAPSHOT_KEY` and export the matching value. VM rejects requests without `?authKey=` when `-snapshotAuthKey` is set.
- **vmauth bearer token** ‚Äî set `bearer_token_env_var: VM_TOKEN`. The component sends `Authorization: Bearer <token>` on every request.
