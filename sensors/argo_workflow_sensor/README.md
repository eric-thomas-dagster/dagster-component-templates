# argo_workflow_sensor

## Purpose

Polls the [Argo Server](https://argo-workflows.readthedocs.io/) REST API for the status of an Argo Workflow and fires a Dagster `RunRequest` (plus optional `AssetMaterialization` / `AssetObservation`) when the workflow reaches a terminal success phase (`Succeeded` or `Skipped`).

Two modes — set exactly one:

- **`workflow_name`** — watch one specific Workflow by name (useful when Dagster knows the name, e.g. it triggered it via `argo_workflow_trigger`).
- **`workflow_template_label`** — watch the most-recently-completed workflow matching a label selector (useful when Argo's own scheduler creates workflows on its own cadence — Dagster reacts to whichever finishes last).

## Example YAML

```yaml
type: dagster_community_components.ArgoWorkflowSensorComponent
attributes:
  sensor_name: argo_nightly_done
  workflow_template_label: workflows.argoproj.io/workflow-template=nightly-aggregation
  namespace: argo
  host_env_var: ARGO_SERVER_URL
  token_env_var: ARGO_TOKEN
  job_name: downstream_processing_job
  asset_key: argo/etl/nightly_aggregation
```

## Workflow phases

Argo's terminal phases (`status.phase` field on a Workflow):

| Phase | Treated as |
|---|---|
| `Succeeded` | Success → fire RunRequest + emit materialization |
| `Skipped` | Success (workflow opted out — counts as completion) |
| `Failed` | Terminal failure → skip |
| `Error` | Terminal failure → skip |
| `Omitted` | Terminal failure → skip |
| `Pending` / `Running` | In progress → skip (will check again next tick) |
