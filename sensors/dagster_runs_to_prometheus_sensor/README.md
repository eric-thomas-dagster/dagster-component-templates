# dagster_runs_to_prometheus_sensor

Push **Dagster's own run lifecycle metrics** (duration, count, last-finished timestamp) to a Prometheus-compatible endpoint on every run completion. Operates as a `@run_status_sensor` pair (one for SUCCESS, one for FAILURE) and POSTs Prometheus text-format on each event.

Targets:

| target_kind | Endpoint used | Works with |
|---|---|---|
| `push_gateway` (default) | `POST {base}/metrics/job/{push_job}/dagster_job/{job_name}` | Prometheus Push Gateway, VictoriaMetrics-via-pushgateway-mode |
| `vm_import` | `POST {base}/api/v1/import/prometheus` | VictoriaMetrics directly |
| `remote_write` | `POST {base}/api/v1/push` | VictoriaMetrics remote-write (NB: Mimir/Cortex require snappy-compressed protobuf — see "Mimir caveat" below) |

This is the **operational telemetry plumbing**, distinct from the data-plane `dataframe_to_prometheus` sink. Customers who already run Prometheus + Grafana for app-side metrics can graph Dagster job latency / failure rates in the same dashboards.

## Metrics pushed

Three series per run-status event:

```text
# HELP dagster_run_duration_seconds Run duration in seconds.
# TYPE dagster_run_duration_seconds gauge
dagster_run_duration_seconds{job_name="ingest_orders",status="success",location="prod-east"} 42.7

# HELP dagster_run_count Cumulative count of completed runs by status.
# TYPE dagster_run_count counter
dagster_run_count{job_name="ingest_orders",status="success",location="prod-east"} 1

# HELP dagster_run_last_finished_timestamp Unix timestamp of last finished run.
# TYPE dagster_run_last_finished_timestamp gauge
dagster_run_last_finished_timestamp{job_name="ingest_orders",status="success",location="prod-east"} 1748534567
```

Labels: `job_name`, `status` (success / failure), `location` (configurable). The push gateway / VM aggregates across pushes.

## Example — Prometheus Push Gateway

```yaml
type: dagster_component_templates.DagsterRunsToPrometheusSensorComponent
attributes:
  sensor_name: dagster_runs_to_prometheus
  base_url_env_var: PROM_PUSHGATEWAY_URL
  target_kind: push_gateway
  push_job: dagster_runs
  location_label: prod-east
```

Export `PROM_PUSHGATEWAY_URL=http://prometheus-pushgateway.svc:9091`.

## Example — filter to specific jobs

```yaml
type: dagster_component_templates.DagsterRunsToPrometheusSensorComponent
attributes:
  sensor_name: dagster_runs_to_prometheus_critical
  base_url_env_var: PROM_PUSHGATEWAY_URL
  monitor_jobs: [order_pipeline, billing_pipeline]
```

Filters to two named jobs. Default = all jobs.

## Mimir caveat

Mimir / Cortex's distributors require **snappy-compressed protobuf** on `/api/v1/push`, not text format. The cleanest pattern for Mimir at scale is to put a Prometheus Push Gateway in front of Mimir (Mimir scrapes the gateway), and point this sensor at the gateway. Direct-to-Mimir support via the snappy/protobuf encoding is on the v2 roadmap.

## Why two sensors

Dagster's `@run_status_sensor` decorator filters on a single `run_status`. The component wires up two — one for SUCCESS, one for FAILURE — so both event types push their own metric line with the right `status` label.
