# dagster_runs_to_splunk_hec_sensor

Push **Dagster run lifecycle events** (success + failure) directly to Splunk via the HTTP Event Collector (HEC). Splunk-native wire format, single hop — no OTel Collector required.

Operational telemetry. Parallels the existing `dagster_runs_to_prometheus_sensor` and `dagster_runs_to_influxdb_sensor` — same `@run_status_sensor` pair, different transport. If you'd rather route through an OTel Collector (e.g. you already operate one for app traces/metrics), use [`dagster_runs_to_otlp_sensor`](../dagster_runs_to_otlp_sensor/) — it works against Splunk via the Splunk Distribution of OpenTelemetry Collector.

For failure-only Splunk HEC shipment with richer error context, you can also stack `dagster_failures_to_otlp_sensor` → Splunk OTel Collector or use this same sensor with `monitor_jobs`.

## What lands in Splunk per run-status event

```json
{
  "event": "Dagster run abc-123 for job ingest_orders success (42.7s)",
  "sourcetype": "dagster:run_event",
  "source": "dagster",
  "host": "prod-east",
  "index": "dagster",
  "fields": {
    "dagster_run_id": "abc-123-uuid",
    "dagster_job_name": "ingest_orders",
    "dagster_status": "success",
    "dagster_duration_seconds": 42.7
  }
}
```

Failures add `dagster_failure_step` to `fields` and append `step=… err=…` to the `event` body.

Query in Splunk Web:

```spl
// All failures in the last hour with step + error context
search index=dagster dagster_status=failure earliest=-1h
| table _time, dagster_job_name, dagster_failure_step, _raw

// p95 run duration per job
search index=dagster dagster_status=success
| stats perc95(dagster_duration_seconds) AS p95 by dagster_job_name

// Failure rate
search index=dagster sourcetype=dagster:run_event
| stats count by dagster_job_name, dagster_status
| chart count by dagster_job_name dagster_status
```

## Example

```yaml
type: dagster_component_templates.DagsterRunsToSplunkHecSensorComponent
attributes:
  sensor_name: dagster_runs_to_splunk
  hec_url_env_var: SPLUNK_HEC_URL
  hec_token_env_var: SPLUNK_HEC_TOKEN
  index: dagster
  sourcetype: dagster:run_event
  source: dagster
  location_label: prod-east
  verify_ssl: true
```

Required env:

```
SPLUNK_HEC_URL=https://splunk.acme.com:8088/services/collector
SPLUNK_HEC_TOKEN=<your_hec_token>
```

## Setup checklist

1. **Configure HEC on the Splunk side** — Splunk Web → Settings → Data inputs → HTTP Event Collector. Enable globally, create a token, assign default index.
2. **Set the env vars** on every machine running Dagster's daemon (`SPLUNK_HEC_URL`, `SPLUNK_HEC_TOKEN`).
3. **Add the YAML** to your project's `defs/` folder.
4. **Toggle the sensors on** — `dg dev` UI, or set `default_status: RUNNING` to auto-enable.

## Relationship to the Splunk compute log manager

Distinct concerns:

| | This sensor (`dagster_runs_to_splunk_hec_sensor`) | `compute_log_managers/splunk/SplunkComputeLogManager` |
|---|---|---|
| What ships | Run start/success/failure events | Op stdout/stderr (every captured log line) |
| When | At run completion | At step finish |
| Layer | Project (`defs/`) | Instance (`dagster.yaml`) |
| Splunk side | One event per run | One event per stdout/stderr line |

Use both side-by-side: the CLM gives you the granular op output (debugging), this sensor gives you the run-level rollups (SLO monitoring, failure alerting).

## Relationship to the audit-log sinks

`audit_logs_to_splunk` (in `assets/sinks/`) pulls audit events from Dagster+ GraphQL (or similar source) and writes Pandas DataFrames to Splunk HEC on a schedule. This sensor reacts in real time to Dagster's own run-status events. **Complementary** — audit-log sink is the historical / compliance record (scheduled), this sensor is the operational signal (real-time).
