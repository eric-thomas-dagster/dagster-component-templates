# Firebase Storage Monitor

Watch a Firebase Storage bucket for new files and trigger jobs on detection.

Firebase Storage is Google Cloud Storage under the hood, so the deduplication
model is the same: cursor-based on the object's `updated` timestamp, keyed
per-run by `<bucket>/<name>-<etag>`.

## Installation

`pip install firebase-admin`

Pair with a `firebase_resource` component in the same Dagster project.

## Configuration

```yaml
type: dagster_component_templates.FirebaseStorageMonitorComponent
attributes:
  sensor_name: firebase_uploads_sensor
  bucket_name: my-project.appspot.com
  resource_name: firebase_resource
  prefix: uploads/
  blob_pattern: ".*\\.(png|jpg|jpeg|webp)$"
  job_name: process_uploaded_images
```

## Run Config Schema (non-partitioned mode)

Each RunRequest carries the object metadata in `run_config`:

```python
{
  "ops": {
    "config": {
      "bucket": str,
      "name": str,
      "size": int,
      "etag": str,
      "updated": str,          # ISO 8601
      "content_type": str,
      "prefix": str,
    }
  }
}
```

## Dynamic partition mode

Set `dynamic_partition_name` to register each new blob name as a partition
under that name. Downstream partitioned assets can pick up the partition key
directly.

```yaml
dynamic_partition_name: firebase_uploads
```

## Authentication

Handled by the paired `firebase_resource` component. The service account used
needs `roles/storage.objectViewer` (or higher) on the target bucket.

## Bucket naming

Firebase Storage buckets are typically `<project>.appspot.com`. If you leave
`bucket_name` empty, the sensor falls back to the FirebaseResource's default
bucket (which itself defaults to `<project_id>.appspot.com`).
