# Precisely Job Sensor

Fires a Dagster `RunRequest` when a [Precisely Connect ETL](https://www.precisely.com/product/precisely-connect/connect) job-run reaches terminal SUCCESS. Precisely owns the schedule and execution; Dagster polls the documented Job Status endpoint and reacts.

## What you CAN do

| Capability | Verified path |
|---|---|
| Watch a known Precisely run-id | `GET {host}/projects/{jobRunId}/status` ŌĆö the **only** Precisely Connect ETL REST endpoint documented publicly. Returns a plain-text status string. |
| Fire a Dagster `RunRequest` on terminal SUCCESS | When status is `COMPLETED` or `COMPLETED_WITH_WARNINGS`, the sensor emits a RunRequest for `job_name` and dedupes by `run_id` |
| Skip on terminal failure | When status is `COMPLETED_WITH_ERRORS` / `CANCELLED` / `ERRORED` / `LOST_CONTACT`, the sensor skips with a reason rather than firing |

## What you CANNOT do (and why this component doesn't pretend to)

| Wished-for capability | Reality |
|---|---|
| Submit / trigger a Precisely run from Dagster | Precisely does **not** publish a submit endpoint. Earlier versions of this integration shipped a best-guess `POST /projects/{job_id}/run` that 404'd in real customer installs ŌĆö it has been removed. |
| Discover the latest run-id for a given job_id | Precisely does **not** publish a list-runs endpoint. An earlier "Mode B" used a best-guess path; that has also been removed. You need the `job_run_id` from your Precisely-side scheduler / UI / handoff. |

## What this does on each tick

1. Reads `host` + token from env vars (or a shared `PreciselyResource`)
2. Calls `GET /projects/{job_run_id}/status` ŌĆö the documented endpoint
3. If status is `COMPLETED` or `COMPLETED_WITH_WARNINGS` and `run_id` differs from the cursor ŌåÆ emit `RunRequest` for `job_name`
4. Update the cursor so the same completion never fires twice

## Example

```yaml
type: dagster_community_components.PreciselyJobSensorComponent
attributes:
  sensor_name: precisely_etl_done
  job_run_id: "abc-123-def-456"
  host_env_var: PRECISELY_HOST
  api_token_env_var: PRECISELY_API_TOKEN
  job_name: downstream_processing_job
  minimum_interval_seconds: 60
  default_status: running
```

## Connection: env vars OR resource

| Option | YAML |
|---|---|
| Env vars (simplest) | `host_env_var: PRECISELY_HOST` + `api_token_env_var: PRECISELY_API_TOKEN` |
| Shared resource | `resource_key: precisely` (then wire a `PreciselyResource` in `definitions.py`) |

The `PreciselyResource` class is exported from `dagster_community_components` if you want a shared instance across multiple sensors.

## Status enum

Per the [public REST docs](https://help.precisely.com/r/Connect-ETL/pub/Latest/en-US/Connect-ETL-Rest-API-Reference/Job-Status):

| Category | Statuses | Behavior |
|---|---|---|
| **Terminal SUCCESS** | `COMPLETED`, `COMPLETED_WITH_WARNINGS` | Fires `RunRequest` |
| **Terminal FAIL** | `COMPLETED_WITH_ERRORS`, `CANCELLED`, `ERRORED`, `LOST_CONTACT` | Skips with reason |
| **In-progress** | `WAITING`, `RUNNING` | Skips with status |

## Run config passed to the triggered job

```yaml
ops:
  config:
    precisely_job_run_id: <the run id that completed>
    precisely_status: <COMPLETED or COMPLETED_WITH_WARNINGS>
```

Read in your op:

```python
@op
def downstream_op(context):
    run_id = context.op_config["precisely_job_run_id"]
    status = context.op_config["precisely_status"]
    ...
```

## Validation level

`code` ŌĆö `dg check defs` passes and the polled endpoint matches Precisely's public REST docs character-for-character. **End-to-end validation against a live Precisely Connect ETL cluster has not been performed** (we don't have one to validate against). Treat the live behavior as best-effort until a customer confirms a working RunRequest fire.
