# Snowflake Snowpipe Load Sensor

Fires a Dagster `RunRequest` each time a Snowflake **PIPE** ingests one or more new files. Reads file-level events from `INFORMATION_SCHEMA.COPY_HISTORY` — the same source the Snowsight Pipe Activity UI uses.

## Purpose

Snowpipe is Snowflake's continuous-ingestion model. There are two flavors:

| Mode | How ingestion fires | Where Dagster fits |
|---|---|---|
| **`AUTO_INGEST = TRUE`** | Cloud-storage event (S3 PUT / GCS object PUT / Azure blob create) → SNS/SQS → Snowflake auto-COPYs the file | Dagster reacts AFTER the COPY lands — this sensor triggers downstream assets per file |
| **`AUTO_INGEST = FALSE`** | `ALTER PIPE … REFRESH` (Dagster materializes the snowpipe asset, which calls REFRESH) | Same sensor watches the resulting loads |

In both cases this sensor is the "Dagster reacts to a Snowflake-side event" piece — file-level granularity, not just table row count.

## Differentiator vs `snowflake_table_observation_sensor`

| Aspect | `snowflake_table_observation_sensor` | `snowflake_snowpipe_load_sensor` (this) |
|---|---|---|
| What it watches | Row count on a table | File-level loads on a specific PIPE |
| Granularity | Row delta since last poll | One event per ingested file |
| Metadata exposed | `row_count`, `last_change_time` | `file_name`, `file_size`, `row_count`, `status`, `error_count`, `first_error` |
| Fires per | Row-count change (one RunRequest) | File ingested (one RunRequest per file) |
| Best when | You don't care which file, just "did rows arrive" | You want per-file orchestration / metadata |

Use both if you want layered orchestration: row-count sensor for cheap polling, snowpipe sensor for rich per-file events.

## Example

```yaml
type: dagster_component_templates.SnowflakeSnowpipeLoadSensorComponent
attributes:
  sensor_name: orders_pipe_load_sensor
  job_name: process_new_orders_job
  pipe_name: ORDERS_PIPE
  destination_table: ORDERS_INGESTED

  account: "{{ env('SNOWFLAKE_ACCOUNT') }}"
  user:    "{{ env('SNOWFLAKE_USER') }}"
  database: DAGSTER_DEMO
  schema:   STAGING
  warehouse: COMPUTE_WH

  authenticator: SNOWFLAKE_JWT
  private_key_file: "{{ env('SNOWFLAKE_PRIVATE_KEY_FILE') }}"

  minimum_interval_seconds: 60
  pass_file_metadata: true
```

## Run config passed to the triggered job

When `pass_file_metadata: true` (the default), each `RunRequest` carries this `run_config`:

```yaml
ops:
  config:
    snowpipe_file_name:      <S3/GCS/Azure object key>
    snowpipe_last_load_time: <ISO timestamp>
    snowpipe_row_count:      <int>
    snowpipe_file_size:      <bytes>
    snowpipe_status:         <Loaded | Load failed | …>
```

Read in your op:

```python
@op
def process_new_file(context):
    file_name = context.op_config["snowpipe_file_name"]
    row_count = context.op_config["snowpipe_row_count"]
    ...
```

## Cursor + dedup

The sensor uses the `LAST_LOAD_TIME` of the most-recently-seen file as its cursor. Each `RunRequest` is issued with `run_key=snowpipe:<pipe>:<file_name>`, so:

- The same file never triggers the downstream job twice, even if multiple sensor ticks see it in their lookback window
- Re-deploys / cursor resets re-process at most `lookback_minutes` of history
